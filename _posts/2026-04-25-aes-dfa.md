## 前置工作

通过 USB-C 直连电脑，可以同时满足供电和串口通信。然后串口波特率配置为 `38400` 即可看到题目信息。

![](/assets/img/posts/aes-dfa/1.png)

开发板重要引脚和接口如下。

![](/assets/img/posts/aes-dfa/2.png)

通过串口发送 `AES` 后，芯片会执行 AES-128 加密运算，触发信号会在加密过程中拉高。本实验以电压故障注入为例，在 AES 第 9 轮运算附近注入故障，产生足够多的错误密文，再通过 DFA 恢复密钥。

## 逻辑接线图

下图展示了电压故障注入攻击的硬件接线方案，核心设备是 PowerShorter。

![](/assets/img/posts/aes-dfa/3.png)

> 整体攻击流程：串口触发 AES 加密运算 → 输出触发信号 → PowerShorter 产生短路毛刺 → AES 运算出错 → 读取错误密文 → 结合正确密文进行 DFA。

## 编写故障注入代码

```python
from power_shorter import *
import faultviz
import random
import serial
import time
import numpy as np
from osrtoolkit.dfa import aes as dfaaes

# 初始化串口和 PowerShorter
ps = PowerShorter("COM4")
ser = serial.Serial("COM19", 38400, timeout=0.5)

# 启动实时可视化界面服务
faultviz.start_view_service(port=12345)
vt = faultviz.ViewWidget()

# 初始化列表，用于收集满足 DFA 攻击条件的有效密文对
fault_list_pair = []

# 获取一次正常加密结果，作为后续 DFA 攻击的参考基准
ser.reset_input_buffer()
ser.reset_output_buffer()
ser.write(b"AES\n")
c_correct = bytes.fromhex(ser.read(32).decode())

# 定义重启函数
def restart():
    ps.relay(RELAY.RELAY2, 1)
    time.sleep(0.5)
    ps.relay(RELAY.RELAY2, 0)
    time.sleep(0.5)

# 执行一次完整的电压毛刺故障注入尝试
def STM32F303_attack():
    glitch_pulse = random.randint(14, 18)
    glitch_delay = random.randint(229500, 232500)
    ps.engine_cfg(Engine.E1, [(0, glitch_delay), (1, glitch_pulse), (0, 1)])
    ps.arm(Engine.E1)
    time.sleep(0.5)

    ser.reset_input_buffer()
    ser.reset_output_buffer()
    ser.write(b"AES\n")

    try:
        c = bytes.fromhex(ser.read(32).decode())
        s = ps.state(Engine.E1)
        if s == "glitched":
            if dfaaes.DfaAESSubFunctions.check_fault_pair(c_correct, c) == 9:
                status = "SUCCESS"
                fault_list_pair.append((c_correct.hex(), c.hex()))
            else:
                status = "NORMAL"
            vt.update(
                state=status,
                delay=glitch_delay,
                pulse=glitch_pulse,
                result=c.hex(),
            )
    except:
        restart()

# 展示故障注入结果的实时可视化界面
STM32F303_attack()
vt.show()

# 反复尝试故障注入，直到收集到足够多的有效故障对
while 1:
    STM32F303_attack()

# 用收集到的故障对反推 AES 密钥
DFA_KEY = dfaaes.DfaAES(fault_list_pair)
DFA_KEY.get_master_key()
```

### 为什么要在 AES 的第 9 轮注入故障

- AES-128 共有 10 轮，其中第 10 轮会省略 MixColumns，这使第 9 轮成为理想的故障注入位置。
- 注入过晚时，故障只影响少量字节，无法提供足够的求解约束；注入过早时，故障会扩散到大量字节，差分关系又会过于复杂。
- 在第 9 轮注入单字节故障后，MixColumns 会将错误扩散到同一列的 4 个字节；第 10 轮不再执行 MixColumns，因此最终密文保留了结构清晰的 4 字节差分。
- 通过对比正确密文和多组故障密文，可以逐步缩小最后一轮轮密钥的候选范围，最后再通过密钥扩展的逆运算恢复原始主密钥。

### 判断故障成功的依据

> dfaaes.DfaAESSubFunctions.check_fault_pair(c_correct, c) == 9
{: .prompt-info }

- `c_correct` 是正确密文，`c` 是 Glitch 后获得的待检测密文。
- `check_fault_pair()` 通过比较两者的差分结构判断故障轮次：1 字节差分通常对应第 10 轮，结构化的 4 字节差分对应第 9 轮，大量字节差分则可能来自更早轮次。
- 只有当函数返回 `9` 时，程序才将该故障对视为有效数据。
