## 实验目标

本文以 Nordic nRF52840 为目标芯片，通过电压故障注入干扰 APPROTECT 保护逻辑，尝试重新开启已关闭的 SWD 调试接口，并读取受保护的固件。

nRF52840 基于 Arm Cortex-M4，广泛应用于 USB Dongle、智能穿戴和物联网设备。本实验也是我的故障注入入门实践。

## 环境与工具

- 目标设备：nRF52840 USB Dongle
- 调试器：J-Link
- 烧录工具：J-Flash、nrfjprog
- 调试工具：OpenOCD
- 故障注入设备：PowerShorter
- 辅助设备：示波器

## 固件烧录与调试

### J-Flash

J-Link 的标准 SWD 接线如下。

![](/assets/img/posts/nrf52-vga/1.png){: .normal }

连接 J-Flash 后，将编译生成的 HEX 固件（例如具有闪灯效果的 `blink.hex`）烧录至 USB Dongle。烧录完成后，需要重新上电才能观察固件运行效果。

### nrfjprog

nrfjprog 可以完成芯片擦除、固件烧录以及内存读写等操作。

```bash
# 擦除芯片
nrfjprog -f nRF52 --eraseall

# 烧录并校验固件
nrfjprog -f nRF52 --program blink.hex --chiperase --verify

# 读取内存
nrfjprog -f nRF52 --memrd 0x10001208

# 写入内存
nrfjprog -f nRF52 --memwr 0x10001208 --val 0xFFFFFF00
```

## APPROTECT 读保护机制

![](/assets/img/posts/nrf52-vga/2.png){: .normal }

调试器通过 Debug Port（DP）和 Access Port（AP）访问芯片内部资源。DP 负责调试通信和 AP 选择，AP 则负责访问内存、寄存器等具体资源。

![](/assets/img/posts/nrf52-vga/3.png){: .normal }

nRF52840 使用 UICR 中的 APPROTECT 配置控制调试访问。当保护开启时，AHB-AP 与调试访问路径被阻断，调试器无法直接读取 Flash。正常情况下，只能通过 CTRL-AP 执行 Recover；该操作会擦除芯片内容，因此无法保留原始固件。

![](/assets/img/posts/nrf52-vga/4.png){: .normal }

> 芯片需要在启动阶段读取 APPROTECT 配置并建立保护状态。如果在这一时间窗口内注入故障，使配置读取或后续判断出现异常，就有机会在不擦除 Flash 的情况下恢复 SWD 访问。
{: .prompt-info }

## 电压故障注入实践

### 注入位置

本实验选择 DEC1 作为电压毛刺注入点，通过短暂连接 DEC1 与 GND，将供电电压快速拉低。

![](/assets/img/posts/nrf52-vga/5.png){: .normal }

### 注入时机

以 PowerShorter GPIO 的供电上升沿作为触发基准，通过调节延时，将毛刺定位到芯片启动阶段的保护配置读取窗口附近。

![](/assets/img/posts/nrf52-vga/6.png){: .normal }

延时参数用于补偿 GPIO 开始供电到芯片执行 APPROTECT 判断逻辑之间的时间差。实际搜索时，需要同时扫描毛刺延时和毛刺宽度。

### 实验流程

1. 使用 PowerShorter 的 GPIO 为 USB Dongle 供电，并通过 Python 控制设备重启。
2. 在 DEC1 注入短路毛刺，扫描合适的延时与宽度参数。
3. 使用 OpenOCD 尝试通过 SWD 连接目标芯片。
4. 若能够成功导出 Flash，则说明故障注入绕过成功。

OpenOCD 使用示例如下。

```bash
# 使用 J-Link，以 SWD 方式连接 nRF52
openocd -f interface/jlink.cfg \
  -c "transport select swd" \
  -f target/nrf52.cfg

# 从地址 0x0 开始导出 0x1000 字节
openocd -f interface/jlink.cfg \
  -c "transport select swd" \
  -f target/nrf52.cfg \
  -c "init; dump_image nrf_flash.bin 0x0 0x1000; exit"
```

> Windows 环境下使用 OpenOCD 时，可能需要将 J-Link 驱动切换为 WinUSB。

## 总结

- APPROTECT 由硬件调试访问控制机制实现，不能通过修改普通应用代码直接绕过。
- 电压故障注入的关键是同时确定物理注入点、启动阶段的时间窗口和合适的毛刺宽度。
- 通过软件控制 GPIO 可以建立启动时间基准，再结合示波器波形与参数扫描逐步缩小有效窗口。
- 成功恢复 SWD 访问后，可以通过 OpenOCD 导出 Flash，并进一步开展固件分析。
